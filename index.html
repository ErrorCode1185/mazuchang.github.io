计算机网络和互联网

什么是Internet？

从具体构成的角度

从具体构成角度来说，计算机网络可被分为节点、边以及协议。

节点

节点指的是数以亿计的、互联的计算机设备（也被称为主机、端系统）及主机上运行的应用程序。

边

边指的是通信链路，例如光纤、同轴电缆、无线电、卫星等。从功能上又可划分为接入网链路和主干链路，通信链路的传输速率通常采用比特/秒（bit/s） 或带宽（bps）度量。

  • 接入网链路：主机连接到互联网的链路
  • 主干链路：路由器间的链路
分组交换设备

分组交换设备用于接收和转发分组：从它的一条入通信链路接收到达的分组，并从它的一条出通信链路转发该分组。我们最常见的两种分组交换设备就是路由器和链路层交换机。链路交换机通常用于接入网中，而路由器通常用于端系统到端系统中通信。

协议

协议定义了在两个或多个通信实体之间交换的报文格式和次序，以及在报文传输和/或接收或其他事件方面所采取的动作，简而言之，协议控制发送、接收消息的消息格式，如 TCP、IP、HTTP、FTP、PPP 等。

在 Internet 中所有的通信行为都受到协议制约

从服务角度

从服务角度看，互联网可被分为使用通信设施进行通信的分布式应用和使用通信基础设施为apps提供编程接口（通信服务）

通信设施进行通信的分布式应用

  • web、VoIP、email、分布式游戏系统、电子商务…

通信基础设施为apps提供编程接口（通信服务）

  • 将发送和接收数据的apps与互联网连接起来
  • 为app应用提供服务选择，类似邮政服务
    • 无连接不可靠服务 （UDP）
    • 面向连接的可靠服务（TCP/IP）

网络边缘

网络边缘指的是主机（端系统），及其上面运行的应用程序，如Web、email。它们需要依赖于下层的通信基础设施完成通信，处于网络的边缘。它们是互联网存在的理由（互联网的诞生就是为了主机/应用程序通信服务的）

架构模式

客户/服务器模式（CS）

  • 客户端像服务器请求、接收服务
  • 如Web浏览器/服务器：emil客户端/服务器

对等（peer-peer）模式（P2P）

  • 很少（甚至没有）专门的服务器
  • 如Gnutella、KaZoa、Emule

通信服务

网络边缘采用 TCP/UDP 作为端系统之间的传输协议

TCP - 传输控制协议【RFC793】

TCP - 传输控制协议（Transmission Control Protocol）是一种面向连接的服务，客户端每次发起请求，都要先跟服务端进行握手（在数据传输之前做好准备）

  • 可靠地、按顺序的传送数据

    服务端确认和重传

  • 流量控制

    发送方不会淹没接收方

  • 拥塞控制

    当网络拥塞时，发送方降低发送速率

TCP 适用于HTTP（Web）、FTP（文件传送）、Telnet（远程登录）、SMTP（email）等网络应用场景中。

UDP - 用户数据报协议【RFC786】

UDP - 用户数据报协议（User Datagram Protocol）是一种无连接服务，客户端直接请求，服务器直接响应

  • 无连接
  • 不可靠数据传输
  • 无流量控制
  • 无拥塞控制

UDP 适用于流媒体、远程会议、DNS、Internet电话

网络核心

网络核心指的是因特网端系统的分组交换机和链路构成的网状网络。网络的核心部分主要是为网络边缘部分的端系统提供数据交换的

数据传输方式

网络核心部分有两种数据交换的方式，一种是电路交换，另外一种是分组交换。

电路交换

电路交换是一种端到端的资源分配模式。每条线路独享线路提供的所有带宽，每个呼叫一旦建立起来就能保证性能，如果呼叫没有数据发送，被分配的资源就会被浪费，电路交换通常用于传统电话网络。

在电路交换中，网络资源（如带宽）被分成片，每次呼叫前会先为呼叫分配片，如果某个呼叫没有数据，则其资料片处于空闲状态

电路交换网络中的复用

链路中的电路是通过频分复用（Frequency-Division Multiplexing，FDM）或时分复用（Time-Division Multiplexing，TDM ）以及波分复用（Wave-division
multiplexing，WDM）和码分复用（Code-Division Multiple Access，CDMA）

  • 频分多路复用

    频分多路复用的特点是把电路或空间的频带资源分为多个频段，并将其分配给多个用户，每个用户终端的数据通过分配给它的子通路传输。主要用于电话和电缆电视系统

  • 时分多路复用

    时分多路复用的特点是按传输的时间进行分割，将不同信号在不同时间内传送。时分多路复用又包含两种方式：同步时分复用和异步时分复用

  • 波分多路复用

    波分多路复用的特点是对于光的频分复用。利用波分多路复用可以做到用一根光纤来同时传输与多个频率很接近的光波信号

  • 码分多路复用

    码分多路复用的特点是每个用户可在同一时间使用同样的频带进行通信，是一种共享信道的方法。使用码分多路复用的优点是通信在各方面之间不会相互干扰，且抗干扰能力强

电路交换不适合计算机之间的通信

  • 连接建立时间长
  • 计算机之间的通信具有突发性，如果使用线路交换，则浪费的片较多即使这个呼叫没有数据传递，其所占据的片也不能被别的呼叫使用
  • 可靠性不高

分组交换

分组交换指的是以分组为单位存储-转发的方式，网络带宽资源不再分为一个个的片，传输时使用全部带宽，主机之间传输的数据被分为一个个分组，优点是能够实现资源共享按需使用，在传输中，数据包经过各节点，节点都需要先接收到整个分组并存储起来才能转发到下一分组

分组交换的缺点

  • 延迟较大

    由于分组交换在数据被传输到下一个链路之前，整个分组必须由路由器存储并转发，因此在一个速率为R bps的链路，一个长度为L bits 的分组的存储转发延时： L/R s

  • 排队时延

    如果到达速率>链路的输出速率分组将会排队，等待传输

    如果路由器的缓存用完了，分组将会被抛弃（丢包）

分组交换的关键功能
  • 路由：决定分组采用的源到目标的路径（使用路由算法）
  • 转发：将分组从路由器的输入链路转移到输出链路

分组交换适用于突发式数据传输，但过度使用会造成网络阻塞，分组延迟和丢失

分组交换存储-转发

分组交换的存储转发一段一段从远端传到目标端，按照有无网络层的连接可分为数据报网络和虚电路网络

数据报网络
  • 分组的目标地址决定下一跳
  • 在不同阶段路由可以改变

Internet 中采用数据报网络

虚电路网络
  • 每个分组都带标签（虚电路标识，VCID）标签决定下一跳
  • 在呼叫建立时决定路径，在整个呼叫路径保持不变
  • 路由器维持每个呼叫的状态信息

X.25 和 ATM 中采用虚电路网络

数据报(datagram)的工作原理
  1. 在通信之前，无需建立一个连接，有数据就传输
  2. 每一个分组都独立路由（路径不一样可能会失序）
  3. 路由器根据分组的目标地址进行路由

虚电路(virtual circuit)的工作原理

接入网

接入网是指将端系统物理连接到其边缘路由器（edge router）的网络。边缘路由器是端系统到任何其他远程端系统的路径上的第一台路由器。

住宅接入

宽带住宅接入目前由两种最流行的类型是数字用户线（Digital Subscriber Line，DSL）和电缆因特网接入（Cable Internet Access）

DSL接入

DSL接入借助以前的电话线路，利用调制解调器将上网数据调制（调频、调幅、调相位、综合调制）加载音频信号上，在电话线上传输，在局端将其中的数据解调出来；反之亦然

如图所示，一个用户的 DSL 调制解调器使用现有的电话线与位于电话公司的本地中心局中的数字用户先接入复用器（DSLAM）交换数据。家庭的 DSL 调制解调器得到数字数据后将其转换为高频音，以通过电话线传输给本地中心局；来自许多家庭的模拟信号在 DSLAM 处被转换回数字形式

DSL传输速率

DSL 标准定义了多个传输速率，包括 12Mbps 下行和 1.8Mbps 上线传输速率【ITU-1999】，以及 55Mbps 下辖和 15Mbps 上行传输速率【ITU-2006】。因为这些上行速率和下行速率是不同的，所以这种接入被称为是不对称的。实际上取得的下行和上行传输速率也许少于上述速率。

电缆因特网接入

电缆因特网接入顾名思义也就是通过电缆（通常是网线/光纤）接入互联网，与 DSL 利用本地电话线接入的方式不同，电缆网络接入利用有线电视接入，利用光缆将电视头端连接到地区枢纽，再通过同轴电缆接入，因为该系统引用了光纤和同轴电缆，因此也被称为混合光纤同轴系统（Hyberid Fiber Coax，HFC）

电缆接入互联网需要特殊的调制解调器，这种调制解调器称为电缆调制解调器（cable moden），如同 DSL 调制解调器，电缆调制解调器通常是一个外部设备，通过一个以太网端口连接到家庭 PC。在电缆头端，电缆调制解调器端接系统（Cable Modem Termination System，CMTS）与 DSL 网络的 DSLAM 具有类似功能，即将来自许多下行家庭的中的电缆调制解调器发送的模拟信号转换回数字形式。电缆调制解调器将 HFC 网络划分为上行和下行两个信道。

HFC传输速率

与 DSL 一样 HFC 接入也是非对称的，下行信道分配的传输速率通常比上行信道的高。DOCSIS-2.0 定义了高达 42.8Mbps 的下行速率和高达 30.7Mbps 的上行速率。如同 DSL 网络中的情况那样，由于较低的合同数据率或媒体损耗，可能不一定能达到标准规定的最大可用速率。

企业(和家庭)接入

在企业和大学以及越来越多的家庭环境中，使用局域网（LAN）将端系统连接到边缘路由器，局域网技术有很多，以太网依旧是企业或大学等机构采用的较为流行的接入方式其次是无线接入方式。

以太网接入

以太网接入方式如下图所示，以太网用户使用双绞铜线与一台以太网交换机相连，而后以太网交换机或这样相连的交换机网络再与更大的因特网相连

使用以太网接入，用户通常以 100Mbps 或 1Gbps 速率接入以太网交换机，而服务器可能具有 1Gbps 甚至 10Gbps 的接入速率

无线局域网接入

无线局域网接入有两种形式，一种是无线 LANs（802.11b/g Wifi），还有一种就是广域无线接入（3G/4G/5G）

无线LANs接入

在无线 LANs 环境中，无线用户从接入点接收和发送分组，该接入点与企业网相连，企业网再与有线因特网相连。一个无线 LANs 用户通常必须位于接入点的几十米范围内。基于 IEEE-8.0.11 技术的无线 LANs 接入也被称为 Wifi

广域无线接入

广域无线接入由电信运营商提供的基站来发送和接收分组，与Wifi不同的是，一个用户仅需要位于基站的数万米范围内。

Internet结构和ISP

互联网结构

互联网是由千千万万个小型的局域网组成的巨大网络，端系统通过接入ISPs（Internet Service Providers）连接到互联网，例如住宅、公司、大学、移动电信的 ISPs。接入 ISPs 互相之间也是互联的，因此任何两个端系统都可互相发送分组到对方。到如今，构成因特网的“网络的网络”已经演化成一个非常复杂的结构，这种演化通常是由经济和国家策略驱动的，最终形成了我们今天的因特网的网络结构

如图所示，在如今的因特网网络的最中心，是一些为数不多的，充分连接的大范围网络，其分布广、节点有限但是彼此之间有着多重连接，因此互联网是一种松散的层次模型，在互联网中心的是第一层 ISPs （如UUNet, BBN/Genuity，Sprint，AT&T）这些 ISPs 通常负责国家/国际覆盖，速率极高，其通常直接与其他第一层的 ISPs 相连或与大量的第二层 ISP 和其他客户网络相连。

第二层 ISPs 是更小些的（通常是区域性的）ISP，其通常与一个或多个第一层 ISPs相连，也可能与其他第二层 ISP 相连

第三层 ISP 与其他本地 ISP 就是接入网（与端系统最近）

由此我们可以看出，一个分组由发送端到接收端之间需要经过许多的网络，这些网络之间的传输速率，带宽能力等也不尽相同，此外 ISP 之间的连接还涉及到费用结算等经济上的问题。因此许多内容提供商（如：Google，Akamai）可能会部署自己的网络，连接自己的在各地的DC（数据中心），走自己的数据，连接若干 local ISP和各级（包括一层）ISP，将它们的服务、内容更加靠近端用户，向用户提供更好的服务，减少自己的运营支出。

分组延时、丢失和吞吐量

分组延时

在分组从源到目标主机的传输过程中需要经过一系列路由器的存储转发，在此过程中，分组经受的延时主要分为节点处理时延、排队时延、传输时延和传播时延（propagation delay），这些时延总体累加起来就是节点的总时延。

  • 处理时延 - nodal processing delay

    处理时延主要指的是检查分组首部和决定将分组导向何处（比如查路由表等一系列操作）以及检查 bit 级差错（利用差错控制编码检查分组是否出错）等所带来的时间上的损耗，处理时延通常是微妙甚至更低数量级的

  • 排队时延 - queuing delay

    排队时延指的是分组在链路上等待传输的时间，这通常取决于先期到达的正在排队等待向链路传输的分组数量。排队时延通常是毫秒到微妙级的

  • 传输时延 - transmission delay

    传输时延指的是将所有分组的比特发送到链路上所需要的时间，假设分组以先到先服务的方式传输，只有当所有已到达的分组被传输后，才能传输刚到达的分组。假设用 L 比特表示该分组的长度，用 R bps 即（b/s）表示从路由器 A 到路由器 B 的链路传输速率。此时它们的传输时延就是 L/R传输时延耗时通常在毫秒到微妙级

  • 传播时延 - propagation delay

    传播时延指的是分组从路由器 A 到路由器 B 期间传播所需要的时间，假设用 d 表示物理链路的长度，用 s 表示分组在媒体上的传播速度（~2*10^8m/sec）此时传播时延就是 d/s，在广域网中传播时延耗时通常在毫秒级

排队时延和丢包

排队时延是节点时延中最为复杂和最有趣的成分，因为排队时延对不同的分组可能是不同的，因此在表征排队时延时，通常使用统计量来度量，如平均排队时延、排队时延的方差和排队时延超过某些特定值的概率。

假设 a 表示分组到达队列的平均速率（a 的单位是分组/秒，即 pkt/s），R 是传输速率（bps/s），假定所有分组都是 L 比特组成，则比特到达队列的平均速率为 La bps，最后假定该队列无限大，此时比率 La/R 被称为流量强度（traffic intensity），因此如果 La/R > 1 则比特到达队列的平均速率超过从该队列传输出去的速率，此时该队列趋向于无限增长，并且排队时延将趋向无穷大！因此在设计系统时流量强度不能大于1

分组丢失

在路由器缓冲区的分组队列中，当分组到达链路的速率超过了链路输出的能力或分组等待排到队头、被传输时，可能会导致分组丢失，丢失的分组可能会被前一个节点或源端系统重传，或根本不重传（取决于传输层协议）

吞吐量

吞吐量是指在源端和目标端之间传输的速率（数据量/单位时间）

  • 瞬间吞吐量：在一个时间点的速率
  • 平均吞吐量：在一个长时间内的平均值

协议层次和服务模型

网络是一个极为复杂的系统，网络功能非常的繁杂：数字信号的物理信号承载、点到点、路由、rdt、进程区分、应用等，从现实来看，网络由许多构成元素和设备，例如主机、路由器、各种媒体的链路、应用、协议、硬件软件等。

在真正开始对网络体系结构进行讨论前，我们先来了解几个术语：

  1. 服务（Service）

    底层实体向上层实体提供它们之间的通信的能力

    • 服务用户（service user）
    • 服务提供者（service provider）

  2. 原语（primitive）

    上层使用下层服务的形式，高层使用低层提供的服务，以及低层向高层提供服务都是通过服务访问原语来进行交互的（Socket API）

  3. 服务访问点 SAP（Services Access Point）

    上层使用下层提供的服务通过层间接口，下层的一个实体支持着上层的多个实体 SAP 有标志不同上层实体的作用

协议分层

我们知道网络协议采用的是分层的体系结构，采用层次化方式实现复杂的网络功能，可以将网络复杂的功能分层，使得每一层次的功能更明确，每一层都实现了其中一个或一组功能，功能中有其上层可以使用的功能，也就是向上层提供的服务，本层协议实体相互交互执行本层的协议动作，目的是实现本层功能，通过接口为上层提供更好的服务，在实现本层协议的时候，直接利用了下层所提供的服务。因此每层的服务其实都是借助下层服务实现的本层协议实体之间交互带来的新功能（上层可以利用的）+ 更下层所提供的服务。

服务的类型

在网络协议层次中，下层向上层提供的服务又分为面向连接的服务和无连接的服务两种方式

面向连接的服务

面向连接的服务( Connection-oriented Service)，这里的连接（Connection）指的是两个通信实体为进行通信而建立的一种结合，面向连接的服务通信的过程需要经历建立连接，通信，拆除连接这三个步骤。面向连接的服务适用于大数据块的传输，不适合用于小的零星报文的传输，面向连接的服务可以保证分组传输的顺序。

服务类型及场景
  • 可靠的信息流 — 适用于传送页面（可靠的获得，通过接收方的确认）
  • 可靠的字节流 — 适用于远程登录
  • 不可靠的连接 — 适用于数字化声音
无连接的服务

无连接的服务（Connectionless Service）指的是两个对等层实体在通信前不需要建立一个连接，不预留资源甚至不需要通信双方都是活跃，在这种情形下分组传输是不可靠的，分组可能重复也可能失序，无连接的服务适合适合传送零星数据。

服务类型及场景
  • 不可靠的数据报 — 适用于电子方式的函件
  • 有确认的数据报 — 适用于挂号信
  • 请求回答 — 适用于信息查询

服务与协议的区别

  • 服务 - Service

    低层实体向上层实体提供它们之间的通信的能力，是通过原语（primitive）来操作的，服务是垂直方向上的概念（下层向上层提供服务）

  • 协议 - protocol

    对等层实体（peer entity）之间在相互通信的过程中，需要遵循的规则的集合，是水平方向上的概念

本层协议的实现要靠下层提供的服务来实现，本层实体通过协议为上层提供更高级的服务

Internet协议栈

Internet协议栈由五个层次组成：物理层、传输层、网络层、链路层、物理层。

  1. 应用层 - 网络应用
    • 为人类用户或者其他应用进程提供网络应用服务
    • 协议：FTP、SMTP、HTTP、DNS …
  2. 传输层 - 主机之间的数据传输

    • 协议：TCP、UDP
  3. 网络层 - 为数据报从源到目的地选择对应的路由

  4. 链路层 - 相邻网络节点间的数据传输

JVM对象创建及内存分配策略

对象的创建过程

如上图所示，JVM对象创建的主要流程大致可分为以下几个步骤：

类加载检查

虚拟机遇到一条 new 指令时，首先将去检查这个指令的参数是否能在常量池中定位到一个类的符号引用，并且检查这个符号引用代表的类是否已被加载、解析和初始化过。如果没有，将会先执行相应的类加载过程。

分配内存

在类加载检查通过后，接下来虚拟机将为新生对象分配内存。对象所需内存的大小在类加载完成后便可完全确定，为对象分配空间的任务等同于把一块确定大小的内存从 Java 堆中划分出来。

对象内存布局

在 Hotspot 虚拟机中，对象在内存中的布局可以分为 3 块区域：对象头、实例数据和对齐填充：

对象头

Hotspot 虚拟机的对象头主要包括两部分信息，分别是 MarkWord 和 KlassPointer。MarkWord 用于存储对象自身的运行时数据， 如哈希码（HashCode）、GC分代年龄、锁状态标志、线程持有的锁、偏向线程ID、偏向时间戳等。KlassPointer 是类型指针，即对象指向它的类元数据的指针，虚拟机通过这个指针来确定这个对象是哪个类的实例。此外对象头中还有 4bit 用来记录数组对象的长度（只有数组对象才有）。

实例数据

实例数据部分存储的是这个对象真正的有效信息，即我们在程序代码里面所定义的各种类型的字段内容，无论是从父类继承下来的，还是在子类中定义的字段都是实例数据的一部分。

对齐填充

如果学过C语言的同学一定了解过一个内存对齐的概念，在这里的对齐填充其实也是一样，就是所有变量的大小之和必须为最大对齐数的整数倍，在 Hotspot 中这个对齐数被定义为 8。

对齐填充的意义

  1. 平台原因（移植原因）

    不是所有的硬件平台都能访问任意地址上的任意数据的；某些硬件平台只能在某些地址处取某些特定类型的数据，否则抛出硬件异常。

  2. 性能原因

    数据结构（尤其是栈）应该尽可能地在自然边界上对齐。原因在于，为了访问未对齐的内存，处理器需要作两次内存访问；而对齐的内存访问仅需要一次访问。

指针压缩

既然提到了对齐填充，这里再扩展一个小知识点，那就是指针压缩

什么是指针压缩

我们都知道，操作系统是有32位和64位之分的，这个位其实指的就是CPU在内存的寻址能力，因此在32位机器上所能寻址的最大空间就是2的32次方等于4个G的空间，如果要按照一个字节寻址，那么指针就需要32位也就是4个字节 ，同理，64位的操作系统的寻址空间就是2的64次方等于16777216TB，要寻址64位的地址，一个指针就需要64位也就是8个字节。

我们现在的机器基本上都是跑在64位系统上。在64位平台的HotSpot中使用32位指针，内存使用会多出1.5倍左右，使用较大指针在主内存和缓存之间移动数据，占用较大宽带，同时GC也会承受较大压力，因此为了减少64位平台下内存的消耗，在JVM中通过对对象指针的压缩编码、解码方式进行优化，使得JVM
只用32位地址就可以支持更大的内存配置，也就是说在存储的时候存储的是32位的指针，在寻址的时候可以通过这个32位的指针寻址64位的地址，JVM 目前所实现的指针压缩算法只能优化到35位地址也就是32G，因此当堆内存大于32G时，指针压缩就会失效，JVM 会强制使用64位（即8字节）来对 Java 对象寻址。

堆内存小于4G时，不需要启用指针压缩，JVM 会直接去除高32位地址，使用低虚拟地址空间

在 JDK 1.6 update14 开始在64bit操作系统中，JVM 支持指针压缩。启用指针压缩：XX:+UseCompressedOops (默认开启)

划分内存的方式

JVM 有两种划分内存的方式，分别是指针碰撞和空闲列表：

指针碰撞(Bump The Pointer)

假设 JVM 堆中内存是绝对规整的，所有被使用过的内存都被放在一边，空闲的内存放在另一边，已使用与未使用内存之间使用一个指针作为内存分界点的指示器，在这种情况下当有新生对象需要分配内存时，只需要将指针朝着空闲内存的方向偏移与新生对象内存大小相同的距离即可。JVM 默认采用指针碰撞方式分配内存

空闲列表(Free List)

假设 JVM 堆中内存不是绝对规整的，已使用的内存与空闲内存相互交错在一起，那就没有办法进行简单的指针碰撞分配内存了，JVM必须维护一个堆内存的空闲列表，在为新生对象分配内存时必须在空闲列表中寻找一块符合要求的内存空间划分给新生对象。并在成功分配内存后对空闲列表进行更新。

解决并发问题

此外，为了解决内存分配时的并发问题，JVM 同样也采用了两种方式解决，分别是CAS和TLAB：

CAS（compare and swap）

CAS 的全称为 Compare-And-Swap，它是一条CPU并发原语。其功能说简单点就是更新某个值前先比较，CAS是一种“无锁”的实现，通过CAS可以保证在多线程并发中操作共享资源的原子性。这个我们有机会聊到并发编程的时候再详细剖析。在此处为对象分配内存时，虚拟机采用CAS配上失败重试的方式保证更新操作的原子性，从而对分配内存空间的动作进行同步处理。

TLAB（Thread Local Allocation Buffer）

TLAB 的全程为 Thread-Local-Allocation-Buffer，翻译过来就是本地线程分配缓冲，其核心思想是通过把内存分配的动作按照线程划分在不同的空间之中进行，即每个线程在Java堆中预先分配一小块内存。每个线程在自己分配的那一小块内存中为新生对象分配内存。

如图：TLAB 由于是线程私有，所以就避免了并发，性能很高。

  • 通过­ XX: +/­- UseTLAB 参数来设定虚拟机是否使用TLAB

    （JVM 默认开启 TLAB）

  • 通过 ­XX:TLABSize 参数可以指定TLAB大小。

对象的访问定位

在前面通过字节码分析 JVM 内存模型的时候我们提到，在 JVM 创建对象时，会在堆区创建一个 OopDesc 实例来表示这个对象，该对象中包含了类的对象头和元数据信息，并在对象元数据区通过一个 klassPoint 指向方法区的类元信息，也就是该类的 Klass 模型，同时在堆区还会生成一个该类的 MirrorKlass 用于存储 Class 对象（也就是我们通过反射可以获取到的），同时在 Klass 中会通过一个指针指向堆区的 MirrorKlass 对象。这种访问对象的方式也被称为直接指针。此外还有一种句柄访问的方式。

句柄访问

如果使用句柄访问的话，那么 Java 堆中将会划分出一块内存来作为句柄池，reference 中存储的就是对象的句柄地址，而句柄中包含了对象示例数据与类型数据各自的具体地址信息：

在 Java 虚拟机规范中只规定了一个指向对象的引用，并没有定义这个引用应该通过何种方式去定位、访问堆中的对象的具体位置，所以对象访问方式也是取决于虚拟机实现而定的。

在 HotSpot 虚拟机中采用的是直接指针访问方式

初始化

内存分配完成后，虚拟机需要将分配到的内存空间都初始化为零值（不包括对象头）， 如果使用TLAB方式分配内存，这一工作过程也可以提前至TLAB分配时进行。这一步操作保证了对象的实例字段在 Java 代码中可以不赋初始值就直接使用，程序能访问到这些字段的数据类型所对应的零值。

半初始化

在初始化的时候还涉及到一个半初始化的概念，意思就是在new一个对象的时候，当加载完类，在堆中分配好空间，做了初始化零值，但是并没有将堆中地址赋予该对象在栈中的引用时，称为对象半初始化。 此时如果启动了一个线程，该对象的 this 引用就会被新创建的线程共享，从而在对象尚未完全构造之前被新的线程可见，这就是 this 逸出现象。

设置对象头

在这一步会在对象头中设置一些必要的信息，比如前面提到的锁状态标识位，对象HashCode，GC分代年龄等。

执行<init>方法

执行<init>方法，即对象按照程序员的意愿进行初始化。 这一步会为属性赋指定值并执行对象构造方法从而真正创建对象。

对象内存分配流程

前面我们大致介绍了一遍对象的创建过程及内存布局，其中也简单提到了创建对象时内存分配所采用的方式及并发问题的处理，接下来我们来看一下对象的内存分配流程：

上面这张图相信大家都不陌生，在大多数情况下，新生的 Java 对象都会在新生代 Eden 区分配内存，当 Eden 区没有足够空间进行分配时，虚拟机将发起一次 Minor GC，回收 Eden 区和 survivor 区的垃圾对象，在 Minor GC 回收后依旧存活的对象将被一起挪到为空的那块 survivor 区中并将这些存活对象的 age（GC分代年龄）+ 1，每次 Minor GC 过后都会进行一次上述操作，当 GC 分代年龄达到 15（不同垃圾收集器这个值可能不同，CMS默认为 6）时，对象会被晋升到老年代，如果老年代已用空间达到阈值，就会触发 Full GC。

Minor GC / Young GC：新生代GC，指发生在新生代的垃圾收集动作，因为 Java 对象大多都具备朝生夕灭的特性，所有 Minor GC 非常频繁，一般回收速度也比较快

Major GC / Full GC：老年代GC，指发生在老年代的 GC，出现了 Major GC，会经常伴随至少一次的 Minor GC（但非绝对的，在 Parallel Scavenge 收集器的收集策略里就有直接进行 Major GC 的策略选择过程）。Major GC 的速度一般会比 Minor GC 慢 10 倍以上。

在经过上述铺垫之后，我们接下来看看内存分配的具体流程：

对象栈上分配

通常来说 Java 对象是在堆上分配内存的，但是这种分配方式也并非绝对，我们知道当对象没有被引用的时候，需要依靠GC进行回收内存，此时如果对象数量较多的时候，就会给GC带来较大压力，也间接影响了应用的性能。因此为了减少临时对象在堆内分配的数量，JVM 还提供了另一种方式，可以使得对象直接在栈上分配内存，如此依赖该对象所占用的内存空间就可以随栈帧出栈而销毁，就减轻了垃圾回收的压力。

什么是栈上分配

栈上分配指的是 JIT 编译器在编译期间根据逃逸分析的结果，发现如果一个对象并没有逃逸出方法的话（没有被方法外的其他地方引用），就可能被优化成栈上分配。分配完成之后，对象可以继续在调用栈内执行，最后线程结束，栈空间被回收，局部变量对象也被回收。这样就无须进行垃圾回收了。

考虑到对象的内存分配需要一整块连续的空间，而栈内存中能直接存放下一整个对象的空间有限，因此在此处 JVM 还做了另一层优化，当通过逃逸分析确定该对象不会被外部访问，并且对象可以被进一步分解时，此时 JVM 不会创建该对象，而是将该对象成员变量分解若干个被这个方法使用的成员变量所代替，这些代替的成员变量在栈帧或寄存器上分配空间，这样就不会因为没有一整块连续空间导致对象内存不够分配。这种方式我们称为标量替换。

标量（Scalar） ：是指无法再分解成更小的数据的数据。Java中的原始数据类型就是标量。

聚合量（Aggregate） ：标量的对立就是可以被进一步分解的量，而这种量称之为聚合量。Java中的对象就是聚合量，因为它可以分解为其他的聚合量和标量

开启栈上分配及标量替换

在程序运行时，我们可以通过 -XX:+DoEscapeAnalysis 参数来开启逃逸分析，通过 -XX:+EliminateAllocations 开启标量替换。这两个参数在 JDK 1.7 及以后是默认开启的。

大对象直接进入老年代

“大对象”顾名思义就是需要大量连续内存空间的对象，例如字符串，数组等。JVM参数 -XX:PretenureSizeThreshold 可以设置具体多大的对象属于大对象，如果对象超过设置大小会直接进入老年代，不会进入年轻代。如此一来就可以避免为大对象分配内存时的复制操作而降低效率。

PretenureSizeThreshold 参数只对 Serial 和 ParNew 两款垃圾收集器有效，因此如果遇到必须使用该参数的场景，推荐使用 ParNew + CMS 的组合

长期存活的对象将进入老年代

这个我们前面已经提到过了，在对象头中有一个 age 属性，该属性用于记录对象的分代年龄，如果对象在 Eden 出生并经过第一次 Minor GC 后仍然能够存活，并且能被 Survivor 容纳的话，将被移动到 Survivor 空间中，并将对象 age 设为 1。对象在 Survivor 中每存活下一个 MinorGC 周期，age 值就 +1，当它的 age 增加到指定阈值（默认为15岁，CMS收集器默认6岁，不同的垃圾收集器会略微有点不同），就会被晋升到老年代中。对象晋升到老年代的年龄阈值，可以通过参数 -XX:MaxTenuringThreshold 来设置。

对象动态年龄判定

为了能更好的适应不同程序的内存状况，虚拟机并不是永远地要求对象的年龄必须达到 MaxTenuringThreshold 才能晋升老年代，为了使那些长期存活的对象能尽快晋升到老年代，JVM 中设定了一个规则：

当前放对象的 Survivor 区域里，一批对象的总大小大于这块 Survivor 区域内存大小的 50%（可以通过 -XX:TargetSurvivorRatio 指定)，那么此时大于等于这批对象年龄最大值的对象，就可以直接进入老年代了，例如 Survivor 区域里现在有一批对象，年龄1 + 年龄2 + … + 年龄n 的多个年龄对象总和超过了 Survivor 区域的 50%，此时就会把年龄n（含n）以上的对象都放入老年代，这就是动态年龄判定规则。对象动态年龄判断机制一般在 minor gc 之后触发。

空间分配担保机制

为了避免 Full GC 发生得过于频繁。在每次发生 Minor GC 之前，JVM 会先检查老年代可用的连续空间是否大于新生代所有对象的总空间，如果这个条件成立，那么就可以确保 Minor GC 的操作是安全的，也就是说假设此次 Minor GC 过后，就算整个年轻代的对象都要晋升老年代，老年代也有足够空间存放这批对象。如果条件不成立，虚拟机就会检查 HandlePromotionFailure 参数是否设置（JDK 1.8 默认设置），如果有这个参数，虚拟机就会继续检查老年代当前最大可用的连续空间，是否大于历次晋升到老年代对象的平均大小，如果大于，将先尝试进行一次 Minor GC，如果小于或不启用该参数，这时将直接进行 Full GC。

Java 的跨平台特性

总所周知 Java 是一种具有“跨平台”特性的高级语言，通俗的来说就是通过 Java 编写出的应用程序，只需要编写一次代码，就可以在不同的系统上运行，开发者无需关心不同操作系统间机器码和指令集上的差异。这使得 Java 可以摆脱了硬件平台的束缚，真正实现了“一次编写，到处运行”的理想。

而 Java 之所以能实现这一点，都得益于 JVM 虚拟机的设计。相信每一个 Java 初学者刚开始都曾通过命令行运行过 Java 程序，我们首先将编写好的代码保存到 .java 文件中，而后需要通过 javac 命令将 .java 文件编译成 .class 字节码文件，之后就可以通过 java 命令运行。而上述步骤中的字节码（ByteCode）就是支撑跨平台特性的基石。JVM 虚拟机通过解释字节码命令的方式执行应用程序，如果想要在不同操作系统下执行相同的字节码则只需下载对应操作系统版本的 JVM 即可。甚至于字节码本身，也可以由其他语言自主编译实现，只需符合 JVM 虚拟机的规范，都可以在 JVM 虚拟机上运行，如Clojure、Groovy、JRuby、Jython、Scala等等。

Java 类加载机制

前面我们说到字节码是支撑 Java 跨平台特性的基石，类加载则是 Java 虚拟机把描述类的数据从 .class 文件加载到内存，并对数据进行校验、转换解析和初始化，最终形成可以被虚拟机直接使用的 Java 类型的过程。

类的生命周期

类从被加载到虚拟机内存中开始，到卸载出内存为止，它的整个生命周期包括了：加载、验证、准备、解析、初始化、使用、卸载七个阶段。其中验证、准备和解析三个部分统称为连接，如图所示：

  • 加载：通过一个类的全限定名来获取定义此类的二进制字节流，使用到该类的时候才会加载，在加载阶段会在内存中生成一个代表这个类的 java.lang.Class 对象，作为方法区这个类的各种数据的访问入口。
  • 验证：验证字节码释放符合 JVM 虚拟机的规范
  • 准备：为静态变量分配内存，赋初始值
  • 解析：将符号引用替换为直接引用
  • 初始化：将类的静态变量初始化为指定值，执行静态代码块

类被加载到方法区中后主要包含运行时常量池、类型信息、字段信息、方法信息、类加载器的引用、对应class实例的引用等信息（这个我们在介绍 JVM 内存模型的时候介绍过）。此外 JVM 虚拟机规范没有强制规定什么时候必须加载类，但是严格规定了有且只有四种情况必须立即对类进行“初始化”。

  1. 遇到new、getstatic、putstatic或invokestatic这4条字节码指令时，如果类没有进行过初始化，则需要先触发其初始化。

    生成这 4 条指令的最常见的 Java 代码场景是：使用 new 关键字实例化对象的时候、读取或设置一个类的静态字段（被 final 修饰、已在编译期把结果放入常量池的静态字段除外）的时候，以及调用一个类的静态方法的时候。

  2. 使用 java.lang.reflect 包的方法对类进行发射调用的时候，如果类没有进行过初始化，则需要先触发其初始化。

  3. 当初始化一个类的时候，如果发现其父类还没有进行过初始化，则需要先触发其父类的初始化。

  4. 当虚拟机启动时，用户需要指定一个要执行的主类（包含 main() 方法的那个类），虚拟机会先初始化这个主类。

双亲委派机制的作用

前面我们通过源码的解析已经大致了解的双亲委派机制的实现，接下来我们再总结一下双亲委派机制的作用，这个在面试中也是经常被问到的：

  1. 沙箱安全机制

    保证了 Java 的核心类只能由引导类加载器加载，这样便可以防止核心API库被随意修改（比如你自己写一个 String 类运行最终肯定还是使用的java.lang.String)

  2. 避免类的重复加载

    刚刚我们在源码中已经看到过这一点了，在各类加载器的 loadClass() 进行加载的时候都会判断当前类有没有被父类加载器加载，如果已经加载就会找到这个类直接返回。

  3. 全盘委托机制

    全盘委托就是指当一个类加载器装载一个类时，除非手动指定使用其他类加载器加载，否则该类所依赖及引用的类也由这个类加载器载入。

此外我们也可以“打破”双亲委派机制，也就是使用自定义的类加载器来对目标类进行加载。“打破”双亲委派机制的方式有很多种，其核心思想就是继承 ClassLoader 实现其 findClass()，关于这部分内容感兴趣的同学可以自行找案例运行，本文就不再过多赘述了。

字面量和符号引用

符号引用

符号引用是属于编译原理方面的概念，包含了下面三类常量：

  • 类和接口的全限定名（Fully Qualified Name）
  • 字段的名称和描述符（Descriptor）
  • 方法的名称和描述符

这些符号引用在编译期保存的只是静态信息，也就是说，在 Class 文件中不会保存各个方法和字段的最终内存布局信息，因此这些字段和方法的符号引用不经过转换的话是无法直接被虚拟机使用的。当虚拟机运行时，需要从常量池获得对应的符号引用，再在类创建时或运行时解析并翻译到具体的内存地址之中。只有到运行时被加载到内存后，这些符号才有对应的内存地址信息，这些常量池一旦被装入内存就变成运行时常量池，对应的符号引用在程序加载或运行时会被转变为被加载到内存区域的代码的直接引用。这个过程被为称为动态链接，符号引用转换成直接引用其实主要也是通过对象头里的类型指针去转换直接引用。

字符串常量池

总所周知，字符串 String 是引用类型，在 JDK8 及以前 String 底层采用的是 char[]存储，而到了 JDK9 及以后开始改用 byte[] 加上编码标记节约空间。同时字符串做为最基础也是最常用的数据类型，如果每次使用都需要单独为字符串分配空间，显然是很不合理的。因此 JVM 为了提高性能和减少内存开销，在实例化字符串常量时进行了一些优化：

为字符串单独开辟一块空间，用于缓存字符串，这块空间也被称作字符串常量池。在创建字符串时，如果池中已经包含一个等于此 String 对象的字符串，则返回代表池中这个字符串的 String 对象。否则，将此 String 对象包含的字符串添加到常量池中，并且返回此 String 对象的引用。字符串常量池底层是一个固定大小的HashTable，其本质上保存的是字符串对象的引用。

此外关于字符串常量池的具体位置，在 JDK1.6、JDK1.7、JDK1.8及以后各有不同

  • JDK1.6 及以前

    JDK1.6 及以前在 HotSport 虚拟机上还有永久代概念，运行时常量池被放在永久代，字符串常量池被放在运行时常量池中

  • JDK1.7

    在 JDK1.7 开始逐步放弃永久代，并逐步改为采用本地内存来实现方法区。此时字符串常量池从永久代被分离出来放到了堆中

  • JDK1.8 及之后

    在 JDK1.8 及以后完全废除了永久代概念，改为采用本地内存实现的元空间（Meta-space）来代替永久代。将原本存在永久代的运行时常量池、类型信息放到了元空间中，字符串常量池则依然在堆中
